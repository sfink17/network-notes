Approaches to Congestion Control

  • There are two broad approaches to congestion control
    • End-to-end congestion control
    • Network-assisted congestion control
  • End-to-end congestion control
    • The network layer does not provide explicit support to the transport layer for congestion control purposes.
    • The presence of congestion in the network must be inferred
    • This is necessary because the IP layer provides no feedback to the end systems about network congestion.
    • When there is an indication of packet loss(timeout or a triple duplicate acknowledgement), the TCP will decrease the wondow size to match what is needed.
  • Network-assisted congestion control
    • Routers (which act as a network-layer component) gives feedback to the sender
    • This feedback gives information about the congestion state in the network
    • This feedback doesn't have to be complex. Many times, it is just a single bit indicator of congestions occuring in the link.
    • Network-assisted congestion control was used in IBM SNA and DEC DECnet architectures.
    • Recently, it was also suggested to be used in TCP/IP networks
    • It is also used in ATM available bit-rate congestion control.
    • The feedback may be simple, but it may also be complex.
  • In network-assisted congestion control, there are usually two ways of feeding the information back from the network to the sender
    • Network feeback via a receiver
    • Direct network feedback
  • The direct feedback usually takes the form of a choke packet.
  • This choke packet tells the network that it is congested.
  • The network feedback via a receiver usually happens when a router updates a field in a packet to indicate congestion. This packet then notifies the sender. (It takes atleast a round trip though.)
